# S09 设计确认问题答案

## 问题 1: MVP 范围

**问题**: S09 MVP 应该包含哪些核心功能？

**答案**:
S09 MVP 应该包含以下核心功能：

1. **团队管理**
   - `spawn_teammate`: 创建持久化队友
   - `list_teammates`: 查看团队成员状态
   - 团队配置持久化 (`.team/config.json`)

2. **消息通信**
   - `send_message`: 点对点消息发送
   - `read_inbox`: 读取并清空收件箱
   - `broadcast`: 向所有队友广播消息
   - JSONL 格式的收件箱 (`.team/inbox/*.jsonl`)

3. **生命周期管理**
   - WORKING 状态：队友正在执行任务
   - IDLE 状态：队友完成任务，线程结束
   - 支持重新激活 IDLE 状态的队友

4. **基础工具集**
   - Lead: 9 个工具 (spawn_teammate, list_teammates, broadcast, send_message, read_inbox, bash, read_file, write_file, edit_file)
   - Teammate: 6 个工具 (send_message, read_inbox, bash, read_file, write_file, edit_file)

**不包含在 MVP 中**:
- 复杂的权限控制
- 队友之间的直接依赖关系
- 任务优先级队列
- 消息持久化历史（只保留未读消息）
- shutdown_request/plan_request 等协议（这些是 s11 的内容）

## 问题 2: System Prompt

**问题**: Teammate 的 system prompt 应该如何设计？

**答案**:
Teammate 的 system prompt 应该包含以下要素：

1. **身份定义**
   ```
   You are {name}, a teammate agent with role: {role}.
   You work as part of a team led by the lead agent.
   ```

2. **工具说明**
   - 明确列出 6 个可用工具
   - 强调 send_message 用于向 lead 或其他队友汇报
   - 说明 read_inbox 会在每轮开始时自动检查

3. **工作模式**
   ```
   - You have a maximum of 50 iterations to complete your task
   - Check your inbox at the start of each iteration for new messages
   - Report progress and results back to the lead using send_message
   - When your task is complete, send a final summary to the lead
   ```

4. **限制说明**
   - 不能创建新的队友（无 spawn_teammate）
   - 不能使用 task 工具创建 subagent
   - 不能使用 compact 工具
   - 工作目录限制在项目根目录

5. **协作指导**
   ```
   - Communicate clearly with the lead about your progress
   - If you need help or encounter blockers, message the lead
   - You can message other teammates if needed for coordination
   ```

**简化版本**（适合 MVP）:
```
You are {name}, a teammate agent (role: {role}).
You have 6 tools: send_message, read_inbox, bash, read_file, write_file, edit_file.
Your task: {initial_prompt}
Report your progress to 'lead' using send_message.
Maximum 50 iterations.
```

## 问题 3: 超时处理

**问题**: 如何处理 teammate 执行超时？

**答案**:

**方案 1: 迭代次数限制（推荐用于 MVP）**
```java
public class TeammateLoop implements Runnable {
    private static final int MAX_ITERATIONS = 50;
    
    @Override
    public void run() {
        for (int i = 0; i < MAX_ITERATIONS; i++) {
            // 检查 inbox
            // 调用 LLM
            // 执行工具
            if (shouldStop()) break;
        }
        updateStatus(TeammateStatus.IDLE);
    }
}
```

**优点**:
- 简单可靠
- 不需要额外的超时机制
- 符合 s09 原版设计

**方案 2: 时间超时（可选，用于生产环境）**
```java
Future<?> future = executorService.submit(teammateLoop);
try {
    future.get(30, TimeUnit.MINUTES); // 30 分钟超时
} catch (TimeoutException e) {
    future.cancel(true);
    updateStatus(TeammateStatus.TIMEOUT);
    messageBus.send("system", "lead", 
        name + " timed out after 30 minutes", "SYSTEM");
}
```

**推荐**: MVP 使用方案 1（迭代次数限制），生产环境可以组合使用两种方案。

**配置项**:
```properties
# .env
TEAMMATE_MAX_ITERATIONS=50
TEAMMATE_TIMEOUT_MINUTES=30  # 可选
```

## 问题 4: 线程池配置

**问题**: 应该使用什么样的线程池配置？

**答案**:

**推荐配置（基于 s09 设计）**:
```java
public class TeammateManager {
    private final ExecutorService executorService;
    
    public TeammateManager() {
        this.executorService = Executors.newCachedThreadPool(
            new ThreadFactory() {
                private final AtomicInteger counter = new AtomicInteger(0);
                
                @Override
                public Thread newThread(Runnable r) {
                    Thread thread = new Thread(r);
                    thread.setName("teammate-" + counter.incrementAndGet());
                    thread.setDaemon(true);  // 不阻止 JVM 退出
                    return thread;
                }
            }
        );
    }
}
```

**为什么选择 CachedThreadPool**:
1. **动态扩展**: 根据需要创建新线程，适合不确定数量的队友
2. **自动回收**: 空闲 60 秒后自动回收线程，节省资源
3. **无上限**: 不限制最大线程数（实际受系统资源限制）
4. **Daemon 线程**: 不阻止 JVM 正常退出

**替代方案（如果需要限制并发）**:
```java
// 最多 10 个并发队友
ExecutorService executorService = Executors.newFixedThreadPool(10,
    threadFactory);
```

**关闭处理**:
```java
public void shutdown() {
    executorService.shutdown();
    try {
        if (!executorService.awaitTermination(60, TimeUnit.SECONDS)) {
            executorService.shutdownNow();
        }
    } catch (InterruptedException e) {
        executorService.shutdownNow();
    }
}
```

## 问题 5: Lead Inbox

**问题**: Lead 的 inbox 应该如何管理和检查？

**答案**:

**自动检查机制（推荐）**:
在 `AgentLoop.runAgentCycle()` 开始时自动检查：

```java
public class AgentLoop {
    private MessageBus messageBus;
    
    public void runAgentCycle() {
        // 1. 自动检查 lead 的 inbox
        if (agentRole == AgentRole.PARENT) {
            List<Message> inbox = messageBus.readInbox("lead");
            if (!inbox.isEmpty()) {
                String inboxContent = "<inbox>" + 
                    gson.toJson(inbox) + 
                    "</inbox>";
                chatHistory.add(Map.of(
                    "role", "user",
                    "content", inboxContent
                ));
            }
        }
        
        // 2. 正常的 LLM 调用
        // ...
    }
}
```

**Inbox 格式**:
```json
[
  {
    "type": "MESSAGE",
    "from": "alice",
    "content": "Login API implementation completed. Tests passing.",
    "timestamp": 1714567890123
  },
  {
    "type": "BROADCAST",
    "from": "bob",
    "content": "Found a bug in authentication module",
    "timestamp": 1714567900456
  }
]
```

**关键设计决策**:

1. **自动 vs 手动**
   - ✅ 自动检查：每轮开始时自动注入，lead 无需显式调用 read_inbox
   - ❌ 手动检查：需要 lead 主动调用 read_inbox 工具
   - **推荐**: 自动检查（更符合异步协作模式）

2. **Drain 策略**
   - ✅ 读取后清空：避免重复处理消息
   - 清空后消息丢失，但这符合"已读即处理"的语义

3. **消息优先级**
   - MVP 不区分优先级，按时间戳顺序处理
   - 所有消息（MESSAGE 和 BROADCAST）平等对待

4. **空 Inbox 处理**
   - 如果 inbox 为空，不注入任何内容到 chatHistory
   - 避免无意义的 `<inbox>[]</inbox>` 干扰上下文

5. **并发安全**
   - MessageBus 的 send/read 操作需要文件锁保护
   - 使用 `synchronized` 或 `ReentrantLock` 保护 JSONL 文件操作

**实现示例**:
```java
public class MessageBus {
    private final Path inboxDir;
    private final Map<String, ReentrantLock> locks = new ConcurrentHashMap<>();
    
    public List<Message> readInbox(String name) {
        Path inboxFile = inboxDir.resolve(name + ".jsonl");
        ReentrantLock lock = locks.computeIfAbsent(name, 
            k -> new ReentrantLock());
        
        lock.lock();
        try {
            if (!Files.exists(inboxFile)) {
                return Collections.emptyList();
            }
            
            List<Message> messages = Files.readAllLines(inboxFile).stream()
                .filter(line -> !line.trim().isEmpty())
                .map(line -> gson.fromJson(line, Message.class))
                .collect(Collectors.toList());
            
            // Drain: 清空文件
            Files.write(inboxFile, new byte[0]);
            
            return messages;
        } finally {
            lock.unlock();
        }
    }
}
```

## 总结

这 5 个问题的答案构成了 S09 实现的核心决策：

1. **MVP 范围**: 聚焦核心的团队管理、消息通信和生命周期
2. **System Prompt**: 简洁明确，强调角色、工具和协作
3. **超时处理**: 使用迭代次数限制（50 轮），简单可靠
4. **线程池配置**: CachedThreadPool + daemon 线程，动态扩展
5. **Lead Inbox**: 自动检查 + drain 策略，无缝集成到 agent loop

这些决策确保 S09 实现既简单又实用，为后续的 s10（协议）和 s11（自主性）打下基础。
