# 抖音选品 Agent 设计文档

## 1. 需求背景

### 1.1 需求对话记录

> Q: 我要设计一个选品agent，它只需要向一个已经存在的登录界面发问，然后获取回答加载到自己的ui界面里。
>
> Q: 这个登录界面是什么？
> A: 抖音小店的登录页面。
>
> Q: 怎么获取信息？
> A: 模拟浏览器操作获取信息。
>
> Q: 选品场景？
> A: 抖音带货的商品挑选。
>
> Q: 目标用户？
> A: 老师说要给新手用。
>
> Q: 未来扩展？
> A: 未来可能还要插到另一个 agent 里作为一个功能模块。

### 1.2 核心需求总结

| 项目 | 内容 |
|------|------|
| 产品定位 | 抖音带货选品辅助工具 |
| 数据来源 | 抖音小店后台（通过模拟浏览器登录并操作） |
| 目标用户 | 新手带货主播/达人 |
| 交互形式 | 自有 UI 界面展示选品结果 |
| 扩展要求 | 需要能作为独立模块，未来嵌入其他 agent |

## 3. 模块设计

### 3.1 浏览器自动化模块

负责与抖音小店后台交互，是整个系统的数据入口。

- 技术选型：推荐 Playwright（比 Selenium 更现代，反检测能力更好）
- 核心流程：
  1. 打开抖音小店登录页面
  2. 用户手动扫码/输入账号登录（agent 不存储密码）
  3. 登录成功后，agent 接管浏览器操作
  4. 根据选品条件，自动在选品广场搜索、筛选
  5. 抓取商品数据（名称、价格、佣金、销量、评分等）
  6. 返回结构化数据给核心逻辑层

- 关键注意事项：
  - 登录环节必须由用户手动完成（合规 + 安全）
  - 需要处理抖音的反爬机制（验证码、频率限制）
  - Cookie/Session 管理，避免频繁重新登录
  - 浏览器实例可以隐藏在后台运行（headless 模式，登录后切换）

### 3.2 选品策略模块

面向新手用户，提供简单易懂的选品逻辑。

- 基础筛选维度：
  - 商品类目
  - 佣金比例（新手关注点）
  - 销量区间
  - 好评率
  - 价格区间
- 新手推荐策略：
  - 预设"新手友好"模板（高佣金 + 中等销量 + 高好评）
  - 一键选品功能，降低使用门槛
  - 简单的趋势提示（近期热销品类）

### 3.3 数据解析模块

将浏览器抓取的原始 HTML/JSON 转换为结构化数据。

- 商品数据结构示例：
```json
{
  "id": "商品ID",
  "title": "商品标题",
  "price": 29.9,
  "commission_rate": 20,
  "commission_amount": 5.98,
  "sales_30d": 15000,
  "rating": 4.8,
  "category": "日用百货",
  "shop_name": "店铺名称",
  "detail_url": "商品链接"
}
```

### 3.4 前端 UI

面向新手，界面要简洁直观。

- 核心页面：
  1. 登录引导页 — 引导用户扫码登录抖音小店
  2. 选品工作台 — 设置筛选条件 + 展示结果列表
  3. 商品详情卡片 — 关键数据一目了然
  4. 收藏/导出 — 支持导出 Excel

- 技术选型建议：
  - 轻量方案：Electron + Vue/React（桌面应用，方便集成浏览器）
  - Web 方案：前后端分离，后端控制浏览器实例

## 4. 作为可插拔模块的设计

> 未来需要嵌入其他 agent，所以必须设计好接口边界。

### 4.1 模块化接口设计

选品 Agent 对外暴露统一接口：

```
接口：ProductSelector
├── login()          → 触发登录流程，返回登录状态
├── search(query)    → 按条件搜索商品，返回商品列表
├── getDetail(id)    → 获取单个商品详情
├── recommend(level) → 按难度等级推荐商品（新手/进阶）
└── export(list)     → 导出商品列表
```

### 4.2 集成方式

- 作为独立服务：通过 HTTP API 对外提供能力
- 作为 SDK/模块：直接被其他 agent import 调用
- 建议两种都支持，优先实现 API 方式（解耦更彻底）

## 5. 风险与注意事项

| 风险 | 说明 | 应对 |
|------|------|------|
| 抖音反爬 | 频繁操作可能触发风控 | 控制请求频率，模拟人类操作节奏 |
| 页面结构变更 | 抖音前端更新会导致抓取失败 | 数据解析模块做好容错，关键选择器集中管理 |
| 登录态失效 | Cookie 过期需要重新登录 | 监测登录状态，及时提醒用户 |
| 合规风险 | 自动化操作可能违反平台规则 | 仅做数据查看，不做自动下单等敏感操作 |

---

## 6. 建议的技术栈

| 层级 | 推荐方案 |
|------|---------|
| 浏览器自动化 | Playwright (Python/Node.js) |
| 后端服务 | Python FastAPI 或 Node.js Express |
| 前端 UI | Vue 3 + Element Plus（轻量、新手友好） |
| 桌面打包（可选） | Electron |
| 数据存储 | SQLite（轻量本地存储） |

---

## 7. 开发优先级建议

1. 先跑通：浏览器自动化 → 登录 → 抓取一个商品列表
2. 再做 UI：把数据展示出来，能用
3. 然后优化：选品策略、新手模板、导出功能
4. 最后抽象：模块化接口，为嵌入其他 agent 做准备
